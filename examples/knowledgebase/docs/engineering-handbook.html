Engineering at Acme Corp

This handbook describes how we build software at Acme. It covers the processes, standards, and expectations that apply to every engineer on the team. When in doubt, default to these guidelines; when they don't apply, use good judgement and document your reasoning.

This document is maintained by the Engineering leadership team. Changes are proposed via pull request and require two approvals from senior engineers.

Code Review Process

Code review is a core quality practice at Acme. Every change merged to a production branch must be reviewed by at least one other engineer. The purpose of review is correctness, maintainability, and knowledge sharing — not gatekeeping.

Opening a Pull Request

Before requesting review, the author is responsible for:

  • Ensuring all automated checks pass (CI, linting, type checks, tests)
  • Writing a clear PR description: what changed, why, and how to test it
  • Linking to the related ticket or GitHub issue
  • Keeping the diff focused — one logical change per PR

Draft PRs are encouraged for work-in-progress feedback. Mark a PR "Ready for Review" only when the author considers it mergeable.

Reviewer Responsibilities

Reviewers are expected to provide actionable, specific feedback. Use the following comment conventions:

  • nit: minor style or wording suggestions (non-blocking)
  • suggestion: a better approach worth discussing
  • must fix: a correctness or security issue that blocks merge
  • question: a genuine clarification request, not a block

Approve only when you are confident the change is correct and meets our standards. Use "Request Changes" if there are must-fix items.

Response Time SLA

To keep work flowing, we hold the following response-time targets:

  • First review response within 1 business day of the "Ready for Review" mark
  • Follow-up responses within 4 hours once a review conversation is open
  • Hotfixes targeting production incidents: within 1 hour, any time

If you cannot meet these targets, communicate this in the PR thread and suggest an alternative reviewer.

Deployment Standards

Acme deploys to production from the main branch on a continuous basis. Every merged change is a candidate for immediate deployment. We rely on feature flags, progressive rollouts, and robust monitoring to release safely and quickly.

Pre-Deployment Checklist

The following checks are performed automatically by our CI/CD pipeline before any deployment proceeds:

  • All unit and integration tests pass on the target environment
  • Database migrations apply cleanly against a snapshot of production data
  • Image scan shows no critical CVEs
  • Load test baseline is within 10 % of the previous week's median latency

Engineers may trigger a deployment manually via the #deployments Slack channel by posting /deploy <service> <commit-sha>. All manual deployments are logged with the requester's name and reason.

Release Windows

Routine deployments are preferred during core hours (09:00–17:00 local time for the on-call engineer). The following windows are restricted:

  • Fridays 16:00 onwards — no non-critical releases
  • Public holidays — only hotfixes approved by the on-call lead
  • End-of-quarter freeze (last 3 business days of Q2 and Q4) — finance systems only

Rollback Procedure

If a deployment causes elevated error rates or SLA breaches, the on-call engineer should initiate a rollback immediately without waiting for root-cause analysis. Rollback is a non-event; escalate after the service is restored.

To roll back a service: post /rollback <service> in #deployments. The pipeline redeploys the previous stable image and notifies the team. A post-incident review is scheduled within 48 hours.

Testing Standards

We test at multiple levels: unit, integration, and end-to-end. Each level has a distinct role; none replaces the others. The goal is a fast, high-confidence feedback loop — not 100 % coverage for its own sake.

Unit Tests

Unit tests verify a single function or class in isolation, with all external dependencies mocked. They must be fast (the full unit-test suite must complete in under 60 seconds) and deterministic. Write unit tests for:

  • Business logic and algorithms
  • Edge cases and error paths
  • Parsers and serialisers

Integration Tests

Integration tests exercise multiple components together, including real infrastructure (database, message queue). They run in CI against a Docker-Compose environment that mirrors production. Integration tests may be slow but must be deterministic — use fixed seeds, clean state between runs.

Integration tests are the primary defence against migration-breaking changes. Never mock the database in integration tests; this is a lesson learned from the Q2 2024 incident where a mocked test passed but a production migration failed.

Coverage Requirements

We target the following minimum coverage thresholds, enforced in CI:

  • Core business logic modules: 90 % line coverage
  • API handlers: 80 % line coverage
  • Infrastructure/glue code: 60 % line coverage (judgement-based)

Coverage below threshold blocks merges. Coverage above threshold is not an end in itself — a test that exists only to hit a line but asserts nothing is worse than no test.

On-Call Rotation

Every engineer with more than 6 months at Acme participates in the on-call rotation. The rotation is weekly, Monday to Monday. The primary on-call engineer is the first responder to all pages from PagerDuty. A secondary on-call engineer provides backup if the primary is unavailable.

On-call compensation: engineers who are paged outside business hours receive 1.5× time off in lieu for each hour of active incident work. This is tracked in Workday and approved automatically with PagerDuty evidence. The on-call schedule is published to the team's Microsoft 365 calendar so managers have visibility of who is carrying the pager each week.

Escalation Path

If you cannot resolve an incident within 30 minutes:

  • Page the secondary on-call engineer
  • Notify the Engineering Manager on Slack
  • Open an incident channel: #incident-<date>
  • Bridge a Microsoft Teams war-room call for incidents affecting multiple services
  • Escalate to the VP Engineering if the incident affects more than 10 % of customers
